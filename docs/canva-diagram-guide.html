4 Agent Event Lifecycle

A horizontal timeline of one event from emit → ack. Great for onboarding decks.

Stages (left → right)

  • 1. Emit — agent publishes tool.call
  • 2. Envelope — wrap with version, attempt, timestamps
  • 3. Route — session key → partition
  • 4. Persist — WAL append
  • 5. Deliver — consumer group pulls in order
  • 6. Ack / Retry — success or back to step 2 with attempt+1

Layout Sketch

[Agent] ─► [Envelope] ─► [Router] ─► [WAL] ─► [Topic] ─► [Consumer] 1 2 3 4 5 6 │ ├──► ack ✓ └──► fail → retry
Design move: put a tiny payload card under stage 2 showing real fields — tenant, project, session_id, agent_id, type, attempt. Makes the abstract concrete.

6 WAL & Crash Replay

Two side-by-side states — before crash and after restart — to show durability.

Before Crash (left)

  • Broker in-memory partition with offsets 0..N
  • WAL file mirrors the same entries, offset markers visible
  • Lightning bolt icon over broker (the crash)

After Restart (right)

  • Broker memory = empty
  • Arrow “replay” from WAL → broker
  • Broker re-populated with same offsets, ready to serve

★ Shared Design System (use across all 8 diagrams)

Reuse the same colors, shapes, and arrow styles everywhere — that consistency is what makes a deck feel professional.

Palette

BG #0B0F1A
Panel #1A2236
Clients #7AA2FF
Core #9EF0C5
Storage #FFB86B
Observe #C79BFF
DLQ/Error #FF7A90

Typography

  • Headings: Inter or Space Grotesk, semibold
  • Body: Inter, regular, 14–16px equivalent
  • Code / IDs: JetBrains Mono or Fira Code

Shape Conventions

  • Rounded rectangle → service / component
  • Cylinder → storage (WAL only)
  • Diamond → decision (retry/DLQ)
  • Pill / chip → metric name or URL
  • Dashed border → "planned, not built"

Arrows

  • Solid → data flow (events)
  • Dotted → telemetry (metrics, traces)
  • Double-ended → bidirectional (WAL read/write)
